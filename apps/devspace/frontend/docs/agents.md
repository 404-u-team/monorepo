# Правила для агентов (AI Assistant)

При работе с проектом искусственный интеллект должен следовать следующим правилам и договоренностям, которые были выработаны в ходе работы над компонентами.

## 1. Архитектура и структура кода (FSD)

- **Строгое следование Feature-Sliced Design (FSD)**: весь код разделен на слои (`app`, `routes`, `widgets`, `features`, `entities`, `shared`). (СМОТРИ ./fsd.md)
- Сущности (`entities`) не могут импортировать фичи (`features`), виджеты (`widgets`) или страницы (`routes`). Иерархия импортов идет строго сверху вниз.
- **Публичное API**: каждый слайс содержит файл `index.ts`. Все внешние слои должны импортировать функционал слайса только через него (напр. `import { ProjectCard } from '@/entities/project'`).
- Модель данных (типы, интерфейсы), API-запросы и UI-компоненты слайса разделяются по соответствующим папкам (`ui`, `model`, `api`).

## 2. Стилизация и CSS Modules

- **CSS Modules**: стили для компонентов обязательно пишутся в файлах `*.module.scss` рядом с самим компонентом (например, `ProjectCard.module.scss`).
- **Запрет на хардкод цветов**: любые значения цветов должны использоваться исключительно через CSS-переменные из дизайн-системы (например, `var(--text--primary)`, `var(--bg--main)`). Никаких `#FFFFFF` или `rgb()` напрямую в стилях компонентов. (СМОТРИ ./styling.md)

## 3. Взаимодействие с API (Swagger)

- Описание API находится в `../backend/swagger-docs/swagger.yaml`.
- Интерфейсы TypeScript (`model/`) в точности описывают контракты из Swagger.
- Все файлы API (`api/`) должны использовать заранее настроенный `apiClient` из `@/shared/api/client`.

## 4. Разработка UI компонентов (React)

- **Использование Shared UI**: максимально переиспользуем компоненты из `shared/ui` (например, `Badge`, `Skeleton`, `IconCounter`). Не дублируем базовый UI в слоях выше.
- Компоненты-карточки (например, для проектов или идей) должны уметь работать как обычный `<article>` и как ссылка `<a>` через пропс `to`, чтобы быть универсальными.
- **Скелетоны**: для сложных компонентов с асинхронной загрузкой обязательно создается `Skeleton`-версия (например, `ProjectCardSkeleton`), повторяющая верстку основной карточки для избежания "прыгающего" контента.
- **Ограничение контента**: длинные тексты (title, description) обязательно обрезаются средствами CSS (например, `line-clamp: 2`).

## 5. Storybook

- Для всех компонентов (в частности в `entities`, `features`, `widgets`, `shared`) пишутся сторисы (`*.stories.tsx`).
- Сторисы должны покрывать все важные состояния: `Default`, `Loading`, `Closed`/неактивные состояния, а также поведение с большим объемом текста, если применимо, (`LongContent`), и т.п..
- Асинхронные запросы в Storybook мокаются через `msw` (поставляя мок-данные, имитирующие реальные ответы сервера, включая связи/пользователей).

## 6. TypeScript и Линтинг

- **Полная типизация**: избегаем `any`. Все стейты, пропсы интерфейсы API строго типизированы.
- Игнорирование линтера `// eslint-disable-next-line` допускается только в исключительных случаях с обязательным пояснением причины.
- **Команды**: для команд используем исключительно **Bun** (`bun run lint`, `bun run storybook`). Ошибки линтера должны быть исправлены, а не подавлены. Не используем `npx`/`npm` для скриптов проекта.
- **Проверки TS**: избегаются `!` (non-null assertions) там, где это возможно. Используются правильные сужения типов (type narrowing). Вместо сокращений (вроде `e`, `arr`) используются полные, понятные имена переменных (`error`, `event`, `users`).
